Situación actual

useScreens() (en src/hooks/use-supabase.ts) trae todas las entradas de screens.
DataManagement solo cambia de pantalla al pulsar “Gestionar datos”: guarda selectedScreenId y usa useScreenData(screenId) para mostrar los registros (screen_data).
useScreenData hace un select con screens!inner, de modo que solo devuelve datos si esa pantalla tiene registros en screen_data. No hay una URL pública ni mecanismo de agrupado, paginación o navegación entre pantallas desde fuera del módulo.
Lo que estás pidiendo

Pintar cada pantalla “operativa” (la que gestiona datos) en una URL propia para poder abrirla, por ejemplo: /display/:screenId o /pantalla/:slug.
Esa URL debe reutilizar la lógica existente: seleccionar la pantalla, cargando sus registros de screen_data, sin duplicar el código que ya tienes en Gestión de Datos.
Asegurarte de que los usuarios puedan saltar de una pantalla a otra (con enlaces o un selector) y que las agrupaciones (por screen_group) tengan sentido visualmente.
Qué toca hacer

Crear una vista de sólo lectura (p.ej. src/pages/ScreenDisplay.tsx) que tome screenId de la URL y use useScreenData(screenId) para mostrar la tabla. Puedes reciclar casi toda la parte visual de DataManagement (solo la tabla, sin edición ni importación).
En el router (src/pages/Admin.tsx / src/main.tsx) añade la ruta pública:
<Route path="/screen/:screenId" element={<ScreenDisplay />} />
Esto permite abrir http://.../screen/132b8... en cualquier terminal (incluidas tus Raspberry).
En ScreenList agrega un botón “Ver” que abra esa URL en una nueva pestaña (window.open('/screen/' + screen.id)), manteniendo la lógica de Gestión de Datos intacta.
Si quieres agrupar por screen_group, ScreenDisplay puede mostrar el grupo y botones “Anterior/Siguiente” buscando las pantallas que comparten ese campo.
Por qué no puedes editar color ni lógica

La tabla screens solo tiene columna header_color. Ya lo usamos en ScreenDialog para guardar y mostrar el color.
Si al editar no ves los cambios reflejados, revisa que los campos se hayan normalizado (los arreglos recientes quitan el 400).
Para mantener la lógica actual, no toques las mutaciones (sigue usando supabase.from('screens')). Solo reusa los datos para la vista nueva.
